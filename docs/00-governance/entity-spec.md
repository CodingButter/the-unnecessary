# The Entity System — Universal Spec

> This is the contract for how every canon **thing** is represented. One pattern for
> every noun in the world: a person, a district, a street, a corner, a building, a
> room, a junk drawer, an object. The character profiles under
> `../20-canon/characters/profiles/**` are the first instance of this contract;
> geography is the second; anything we add later (items, factions, events) follows the
> same kit. Nothing here silently overrides canon; it defines how canon is **stored**.

## 1. Core principles

1. **One file per entity.** Every distinct thing in the world is a single markdown file.
2. **Single source of truth.** Every fact lives exactly once, in its natural form:
   structured facts as data (a fenced `yaml` block), prose as prose (the body). Nothing
   is derivable that is also stored.
3. **Fractal and just-in-time.** An entity's *containment depth* is as shallow or as deep
   as the story needs: a corner may have no children today and grow an entire painted
   scene, with a named crack in the curb and its own child files, in Book Two. Infinite
   ceiling, never a requirement. (This governs how far the tree grows **downward**; it is
   independent of how richly each file's own dossier is filled, which §6 and §14 now hold
   to a deduced-rich default.)
4. **Derive, don't store.** Graphs, maps, route tables, "what's on this street," indexes,
   and child-lists are all **generated by walking the files**. They are never hand-kept.
5. **Add downward, never edit upward.** Going deeper on any entity never forces an edit
   to anything above it. This is what makes principle 3 actually free.

## 2. The entity file

Every entity file has three parts:

```
name.md
├─ frontmatter (yaml)   → title, type, status, authority, summary, parent, tags
├─ prose body           → ## sections, hand-edited, the narrative canon
└─ ```yaml edge block   → structured facts + relationships (machine-read)
```

- **Frontmatter** carries the identity and the single **containment** edge (`parent:`).
- **Prose body** is the narrative — description, feel, history. Hand-edited. The asset.
- **A fenced `yaml` block** (` ```yaml ... ``` `) holds the structured edges and hard
  facts. Scripts extract just this block and parse it with a stock YAML parser. Agents
  read the whole page — prose *and* data, both clean. Parsing a block inside a file is
  the same few lines as parsing a whole file, so we keep **one file**, never a sidecar.

Example skeleton:

```markdown
---
title: "412 Perry"
document_type: "entity"
entity_type: "building"
status: "active-canon"
authority: "world-canon"
parent: elis-neighborhood
summary: "..."
---

# 412 Perry

A narrow two-story house with a sagging porch... (prose, as deep as the story needs)

​```yaml
edges:
  addressed-to: { street: perry-rd, between: [perry-and-clyde, perry-and-maple], along: 0.5, side: north }
  owner: rook-eli            # directional; the inverse is derived
​```
```

## 3. Fractal containment (the folder tree)

- An entity is `name.md`. When it needs to hold children, it grows a **same-named
  sibling folder** `name/`. The file is **never** swallowed into the folder, and there
  are **no `index.md` files** — the entity's content always lives in `name.md`.
- Children nest recursively, with no ceiling:
  `greater-detroit/elis-neighborhood/412-perry/kitchen/junk-drawer.md`.
- **The containment edge lives on the child:** `parent: <id>` in the child's frontmatter.
  A parent **never** lists its children. The child-list is *derived* (and is literally
  the contents of `name/`). Therefore **adding or deepening a child never edits the
  parent** — you only ever add downward.
- **Promotion path** (just-in-time, no upward ripple):
  1. A thing is first a passing mention in an ancestor's prose ("a cramped kitchen").
  2. When it earns detail, it becomes its own file `kitchen.md` (with `parent:`).
  3. When it earns children, it grows `kitchen/` beside itself. Promotion at any step
     touches **only the new file**, never its ancestors.
- **Folders express only physical "inside-of" containment.** One home per entity.
- **Cross-check (free validation):** the folder path already encodes the parent (by
  where the file sits) and the file states `parent:` explicitly. The two must agree; the
  validator flags any file whose `parent:` disagrees with its folder home as a misfile.

### Every introduced entity earns a dossier (the three doors now govern depth and locks)

Every entity the world introduces -- named in a bible, brought into a blueprint, or
rendered on the page -- **earns its own file with a full, deduced dossier**: a complete
description and, where it has a life, a timeline, with creativity filling the gaps the
story has not yet reached so the world is **production-ready** (canon airtight enough to
survive multiple books and, eventually, a film adaptation). What creativity fills is
marked **open** (deduced, mutable, never contradicting canon); what canon or approved
prose establishes is marked **locked** (see §14). A dossier is therefore never "empty
noise": the locked subset is the truth the graph must protect, and the open remainder is
honest, labeled deduction the story can sharpen or overwrite at will.

The three doors no longer decide **whether** a file exists. They now govern **depth
priority** -- which entities get their rich dossier first and how deep it goes -- and
**what must be locked** the moment the file is born:

1. **Canon importance.** The Story Bible, plot, or a Decision establishes it as
   significant -- a named device, motif, or load-bearing element. Highest depth priority,
   and its bible-stated facts are born **locked** (`by:` the bible or Master Timeline).
   (Established *in canon*, not merely known by the author: the radio's thematic-seed
   facts lock because `rook-daniel.md` names it the book's seed, **not** because Chapter 1
   mentions it in a simile.)
2. **Blueprint focus.** A blueprint brings it to `sketch` or above. Focus is a **prose**
   dial -- it governs how vividly the *rendered text* paints the entity, not how deep the
   dossier runs (see §8). The two are independent: a `blur`-focus room the prose barely
   names can still carry a full **open** dossier, and a `crisp`-focus object locks only
   the properties the prose actually renders.
3. **On-page attributes.** A chapter renders it with described, contradictable properties
   (an ivory watch face, a torn cushion). Each rendered property is born **locked** to
   that chapter (`by: bN-chN`) once the chapter is approved, **proposed** while it is
   still a draft.

When a thing is introduced it is born in its correct containment folder, with its deduced
description and (if it changes over time) a timeline, its canon-and-prose facts locked and
the deduced remainder open. A bare reference may still simply **link** to an existing
file; authorial foreknowledge that something *will* matter still lives in the blueprint's
setup/foreshadowing rather than being pre-locked. But the entity, once introduced, gets
its dossier -- the doors decide depth and locks, not existence.

## 4. Edges — everything that is not containment

A folder tree can only say *inside of*. Everything else is an explicit **edge** in the
fenced `yaml` block. Reuse and extend the character relationship vocabulary:

- **Directional** (stored once on the dependent end; the inverse is *derived*, never
  written on the other side): `parent`, `father`, `mother`, `guardian`, `employer`,
  `reports-to`, `mentor`, `landlord`, `owner`, `creator-of`, `patient-of`, `residence`
  (person → building), `addressed-to` (building → a position on the street network),
  `connects` is symmetric (below).
- **Symmetric** (written on both ends, reciprocity-checked): `spouse`, `former-spouse`,
  `sibling`, `friend`, `rival`, `adversary`, `colleague`, `neighbor`, `partner`,
  `acquaintance`, and `connects` (a segment connects its two intersections).
- Off-vocabulary relation labels are rejected by the validator; freeform nuance goes in
  prose, not in an edge label.

All graphs — family tree, street network, ownership, residence, containment — are the
**same derive-by-walking machine** reading these edges.

## 5. Derive-by-walking

A file's **folder location is a human browsing concern only**; it never constrains the
graph. The parser walks **every** file, reads the edges, and assembles every graph in
memory — exactly as `scripts/build-relationship-graph.py` already does for 35 character
profiles. Generated, never-hand-edited views (regenerated on demand):

- the street **map** (the network laid out from segments + intersections),
- **route / distance** tables and travel-times (graph paths × a district's withdrawal
  factor — *distance is physics, time is politics*),
- "**what's on** Perry Road" (addresses referencing Perry's segments),
- the **family tree**, ownership and residence views, and folder/containment indexes.

Because location and connection are decoupled, **where a file sits never fights what it
is connected to.** Containment you *browse*; connection you *compute*.

## 6. The scaling template (a ceiling you fill richly, deducing the rest)

Every entity type has a template listing **all** the sections it *could* carry. The old
discipline left most of it blank -- a stub for most entities most of the time, on the
theory that a near-empty file would be noise. That is **reversed.** Every introduced
entity is now filled into a **rich, deduced dossier**: you write the full description and,
where it has one, the timeline, deducing whatever the story has not yet reached. Deduction
is creative but disciplined -- it may **never** contradict a locked fact or a bible (§14)
-- and everything deduced is marked **open** so it stays honest and freely revisable.

- A **deduced dossier** (the new default) = full prose description plus an `open`-marked
  set of facts, edges, and timeline, with the subset established by canon or approved
  prose marked **locked**. Even an entity nobody has entered yet carries a plausible
  deduced interior, labeled open, ready for the scene that arrives.
- A **canon-dense entity** = a dossier whose facts are mostly **locked**, because the
  bibles and approved chapters have established most of them on the page.

The template is still a **ceiling, not a floor** in one sense: no section is forced where
the entity genuinely has none (a fixed room needs no timeline). But within the sections an
entity does carry, the expectation is now **richness, not minimalism.** The validator
continues to enforce **zero-blanks within whatever sections an entity does declare** -- a
declared section may not be empty -- and a dossier's locked-versus-open split must be made
explicit per §14.

## 7. Geography entity types (the first new instance)

The spatial world splits cleanly into two kinds of thing, each handled by the structure
it fits:

**Describable PLACES — ride the folder tree (clean single-parent containment):**

| Type | `parent` | Notes |
|---|---|---|
| `city` | (root / region) | e.g. Greater Detroit |
| `district` | city | feel + withdrawal status in prose; the coarse map layer |
| `building` / `place` | district | home is the district folder; carries an `addressed-to` edge; grows rooms |
| `room` | building | nests under the building, fractal |
| `object` | room (or building) | the junk drawer, the radio on the shelf |

**The street NETWORK — graph only (shared / spanning, not single-parent):**

| Type | Identity | Key data |
|---|---|---|
| `intersection` | a node where streets meet; *describable* (a corner can be painted) | `connects` the streets that meet here |
| `segment` | a stretch of one street between two intersections; *describable* | owns `street`, `from`, `to`, `length_m` **once** (it is the edge) |
| `street` | a named run of segments; *describable* (the character of the street) | derived membership from its segments |

- An **address is a linear reference**, not a coordinate:
  `addressed-to: { street, between: [A, B], along: 0.0–1.0, side }`. Coordinate-free,
  exact, and human-legible ("north side of Perry, halfway between Clyde and Maple").
  Distances between addresses are **derived** (graph path through segments).
- A `building` lives in the **district folder** (containment) yet is positioned on the
  **network** by its `addressed-to` edge. The two facts never collide: folder = which
  district; edge = where on the streets. Each fact has exactly one home.

## 8. Blueprint binding (the world meets the story)

A chapter blueprint — or a scene within it — references entities **at the depth the
prose actually reaches**, and that single act drives three things:

1. **Depth** — reference the level the scene renders: the house, or the room, or the one
   object in the drawer. No deeper than the prose goes.
2. **Focus** — each reference carries a focus level (`blur → sketch → sharp → crisp`).
   Two orthogonal dials: **depth** = *how granular*, **focus** = *how vivid*.
3. **Just-in-time birth** — referencing an entity is what *calls it into being*. If the
   room has no file yet, the blueprint is the reason it gets one (a stub, or deeper if
   the scene paints it). "Make sure both references exist" = the canon files are ensured
   before drafting.

The writer receives, per scene: **this deep, this vivid, here is the canon** (the file
paths). The reviewer checks the prose rendered exactly those entities at those levels,
inventing nothing outside the files. Example scene block:

```yaml
scene: "Eli's kitchen, after midnight"
references:
  - entity: 412-perry/kitchen            # depth: room
    focus: crisp
  - entity: 412-perry/kitchen/junk-drawer/transistor-radio   # depth: object
    focus: sharp
    note: "[behavior-only] it was his father's"
```

## 9. Time and state — entities have lives, not snapshots

The folder tree is a snapshot, but the story moves through time. A couch lovingly
described in `412-perry/living-room/couch.md` may be carried to Jonah's house partway
through the book. **We never move the file.** Moving it would erase the couch's history,
cause exactly the upward churn we banned, and make no sense -- it is the same couch.

Instead, time-varying facts are recorded as a **timeline** in the entity's own fenced
block, keyed to **in-world story date-time, not chapter number.** This is the crucial
distinction between *when something happens in the world* (the state axis) and *when the
reader is told* (the reading order): a flashback chapter is set at an earlier story-date,
so keyed by date it resolves the couch's *earlier* state correctly, where keying by
chapter would hand back the wrong one. The **folder is the entity's home**; the
**timeline is the truth**:

```yaml
home: 412-perry/living-room          # stable; where the file lives, not a live claim
timeline:
  - when: 2053-10-18                  # in-world date (approximate ok: { circa: 2050 })
    set: { located-in: jonah-house/study }
    note: "Jonah takes it after the fight."
  - when: 2053-10-26
    set: { condition: "water-stained, one leg cracked" }
```

- **State at any story-moment is derived:** start from the home/initial values and replay
  the timeline up to that date. "Where is the couch on October 20?" -> Jonah's study.
- **One date axis holds backstory and on-page events alike.** Eli and Nora's divorce (~6
  years before Book One) and Daniel Rook's death (alive -> deceased, circa spring 2050)
  are timeline events at their in-world dates, no different in kind from a couch moving
  mid-book. This is the **same axis as the Master Timeline**; per-entity timelines are its
  slices, and the per-character state files under `docs/60-continuity/` are the first,
  hand-built version of exactly this.
- **Fixed vs movable.** For a room in a building, home and live location coincide and the
  timeline stays empty. For movable things (a couch, a person, an object) the folder is
  the home shelf and the timeline carries the motion -- location, ownership, condition,
  residence, relationships, and **existence** (built/demolished, alive/dead).
- **The blueprint pins each scene to its story-date, so it gets the right state for free.**
  A scene -- even a flashback -- carries an in-world date; the system resolves every
  referenced entity *as of that date* and hands the writer that state, not the t=0 one and
  not the latest one.
- **Time-aware continuity:** the validator and the diff-judge can catch an entity in two
  states at one story-moment, an object used where it no longer is, or a character on the
  page after they have died.
- **Events promote to entities, just-in-time:** a simple change is a `timeline` line; a
  large or multi-entity beat (a fire that damages a building and injures a person) becomes
  its own dated `event` file the affected entities reference, and both "this entity's life"
  and "everything that happened on October 18" are **derived** by walking events.
- **Dates are canonical and machine-sortable.** Every `when` in a timeline and every
  blueprint scene/chapter date is written in **ISO 8601** (`2053-10-03`, or
  `2053-10-03T18:30` when the clock matters), so the resolver orders and compares them
  deterministically. Approximate values stay explicit and still sortable
  (`{ circa: 2050 }`, `{ before: 2053-06-15 }`). A human-readable label ("Friday,
  October 3, 2053") may accompany the canonical field but never replaces it; the existing
  Book 1 blueprints gain a canonical companion date, losing nothing.

## 10. Diff-driven continuity (fearless revisability)

Because **every reference is an explicit edge**, any change to any file has a
**computable blast radius**. This turns continuity from a manual worry into an automatic,
change-scoped process, and it is the property that lets us revise canon *later* to serve
the story without fear:

1. **`git diff`** identifies what changed (which entities, which facts, which prose).
2. The **reference graph** (edges) plus a **prose name-search** identifies the blast
   radius: every file that points at, or talks about, the changed entity.
3. **Structural check (deterministic, cheap, every commit / git hook):** do all
   references still resolve? Do distances still hold? Did a rename orphan an edge? Did a
   segment-length change make an address impossible? Scoped to the blast radius, so it is
   fast even as the world grows.
4. **Semantic check (AI judge, on demand, scoped):** an agent reads the *change* plus
   only the *blast-radius files* and flags prose that now **contradicts** the change —
   distinguishing a true factual contradiction (green walls vs. yellow walls) from a
   legitimate difference of viewpoint or elaboration (one character finds the kitchen
   cozy, another cramped).

The seam is honest: the structural half is deterministic and runs constantly; the
semantic half costs tokens and runs when you make a revision, reading only what the diff
touched. The payoff is the one you named: **make the decision that fits the story better,
anytime, and the diff tells you everywhere that now has to agree.**

### 10a. Prose as a source — backfilling the timeline (the backward direction)

Continuity also runs **backward**: prose is not only a *consumer* of canon, it is a
*source* of it. A drafted or revised chapter is read by an AI that knows the scene's
anchor date (from the blueprint), so it can resolve **relative and implicit** temporal
claims a script never could — "the couch used to live at the Gormans four months ago,"
read in an October 2053 scene, becomes "couch `located-in: gorman-house` until ~June
2053." For each such claim the AI reconciles against the entity's timeline and takes one
of three actions:

- **Consistent** — the timeline already records it. Nothing to do.
- **New** — the timeline lacks it. **Backfill it** with an explicit lock state (§14) so it
  is auditable and re-checkable. A fact stated in *approved* prose is already canon, so it
  is backfilled **locked** with its source (`by: b1-ch7`), recording canon rather than
  inventing it; a fact from a *draft* chapter is backfilled **proposed** with the same
  `by:`, hardening to locked when that chapter is approved and discarded if the draft is
  cut. A purely deduced backfill with no on-page basis stays **open**.
- **Contradiction** — the timeline says something else for that date. Resolve by lock
  state (§14): if the disagreeing entry is **open** (a deduction), approved prose simply
  overwrites it and locks -- a *resolution*, not a conflict. If it is already **locked**
  (a bible fact or an earlier approved chapter), this is a true canon conflict: **flag it
  for the author; never silently resolve it** (per the canon-conflict rule). The
  diff-judge surfaces exactly which prose and which timeline entry disagree, and whether
  that entry was open or locked.

The payoff: the structured timeline becomes **self-populating from the natural act of
writing.** The author writes the evocative line because it serves the scene; the world
model captures it as queryable, dated, validated canon. Structured continuity becomes a
*byproduct* of good prose, not separate bookkeeping.

## 11. Validation (the rails)

A generalized validator enforces, across all entity types:

- **Referential integrity** — every edge target exists.
- **Folder / `parent` cross-check** — `parent:` agrees with the folder home.
- **Containment acyclicity** — no entity is its own ancestor.
- **Network consistency** — each `segment.length_m` is single-source; `from`/`to`
  intersections exist; no two routes imply contradictory distances; every `addressed-to`
  resolves to a real segment with `along` in `[0,1]`.
- **Symmetric reciprocity** — both ends of `connects`, `spouse`, etc. agree.
- **Vocabulary** — relation labels are on-vocabulary.
- **Zero-blanks within declared sections** — stubs may be stubs; declared sections may
  not be empty.
- **Lock-state integrity** — every `locks:` path resolves to a real fact, edge, or
  timeline index; every `locked` / `proposed` item records a `by:` source; no `open` item
  contradicts a `locked` item or a bible (§14).
- **Reveal-tagging respected** — `[open]` / `[reveal: Book N]` / `[behavior-only]`.

## 12. Consistency with the character system

- `../20-canon/characters/profile-spec.md` is now an **instance** of this
  contract: characters are prose-heavy entities whose relationship edges are this edge
  model. People are **not** spatially contained, so a character lives in the character
  area, not the spatial folder tree; their location is a `residence:` edge.
- The existing fenced relationship blocks (currently a custom `relation: id` line format)
  are standardized to valid `yaml` so a **single parser** serves every entity type.

## 13. The extractable process — adding a new entity type

The whole point. Every entity family is the same **seven-part kit**: a **spec** section
(here), a **template**, a shared **parser**, a **validator**, **view generators**, a
**workflow**, and a **context manifest**. To add a type (item, faction, event):

1. Define its identity + edges in §7-style here.
2. Add its scaling template under the entity `_templates/`.
3. Ensure the parser + validator cover its edges (usually already generic).
4. Add a view generator only if it needs one (a map, a timeline).
5. Done — it inherits derive-by-walking, fractal containment, blueprint binding, and
   diff-driven continuity for free.

This is what turns the project from a novel with tooling into a system that builds worlds.

## 14. Lock state — the open / locked lifecycle

Every structured **fact, edge, and timeline entry** in an entity's fenced `yaml` block
carries a **lock state**. This is the safety mechanism that lets §3 and §6 fill entities
with rich, **deduced** dossiers without ever endangering canon: deduction is always
labeled, and the established truth is always protected. Deduce freely *because* the lock
state keeps the deductions honest and overwritable.

**The three states:**

- **`locked`** — established by **approved manuscript prose** OR **hard bible canon** (the
  Story, Character, or Technology Bible, or the Master Timeline). **Immutable.** It records
  `by:` its source (e.g. `b1-ch1`, `master-timeline`, `character-bible`). A locked item
  changes only through a deliberate, logged canon revision, never through a draft or a
  deduction.
- **`proposed`** — asserted only by **draft (not-yet-approved) prose**. It **hardens to
  `locked`** (keeping its `by:`) when that chapter is approved, and is **discarded** if the
  draft is cut. It records the draft source (e.g. `by: b1-ch3`).
- **`open`** — **deduced / creative fill**, or simply never established on the page.
  **Mutable.** It is the honest default for everything §6 deduces, and it **must never
  contradict a `locked` item or a bible.**

**Encoding.** Keep the clean scalar facts, edges, and timeline exactly as they are -- so
the values stay legible and the existing parser still reads them -- and add a parallel
`locks:` map keyed by **dotted path** into the block. The path names the field
(`edges.owner`, `facts.casing_color`) or the timeline index (`timeline.0`):

```yaml
edges:
  owner: okafor-lena
facts:
  capability: continuous-respiratory-support
  casing_color: pale-green
timeline:
  - when: 2053-10-03T23:59
    set: { remote_auth: lost }
  - when: { circa: 2049 }
    set: { installed: true }
locks:
  edges.owner:        { state: locked, by: b1-ch1 }
  facts.capability:   { state: locked, by: b1-ch1 }
  facts.casing_color: { state: open }
  timeline.0:         { state: locked, by: master-timeline }
  timeline.1:         { state: open }
```

**Locked-vs-open heuristic.** An **exact ISO datetime** sourced from prose or a bible →
`locked`; an **approximate** `{ circa | before | after: ... }` date → `open` (a placeholder
the story can later sharpen to an exact, locked value). A value approved prose states **on
the page** → `locked`; a value **invented** to flesh out the file with no on-page basis →
`open`. When in doubt, prefer `open`: an over-locked deduction becomes a future false
conflict, while an open truth merely waits to be confirmed.

**Promotion (the ordinary, frictionless path).** When approved prose renders an item that
was `open` or `proposed`, that item **locks**, stamped `by: bN-chN`. If the prose value
**matches** the open guess, the deduction is simply confirmed. If the prose value
**differs** from the open guess, **the prose overwrites the value and then locks it.** This
is a **resolution, not a conflict** — an `open` item is by definition a mutable placeholder
the story is entitled to overwrite. Likewise, when a chapter is approved, its `proposed`
items harden to `locked` in place.

**Conflict (the rare, must-flag path).** When new prose asserts a value that disagrees with
an item that is **already `locked`** (a bible fact or an earlier approved chapter), this is
a **true canon conflict.** **Flag it for the author; never auto-resolve it**, per the
project's canon-conflict rule: state which item, which authority owns that fact type,
whether approved prose is affected, and a recommended resolution -- then follow the
Development and Canon Guide's process. Do **not** average the two values, and do **not** let
prose silently overwrite a locked fact the way it overwrites an open one. The single
distinction -- *open → overwrite-and-lock* versus *locked → flag the author* -- is the whole
safety model.

**The model is universal.** Lock state applies to **every** entity type, exactly because
the entity system is one pattern for every noun:

- **Characters** — each `edge` (parentage, spouse, employer, residence), each fact on the
  **birth spine** (birth date, birthplace, heritage), and each **biography** beat carries
  its own lock. A heritage stated in the Character Bible is `locked`; a deduced childhood
  detail is `open` until a chapter renders it.
- **Timelines** — locked **per entry** (via the dotted `timeline.N` path), so an exact,
  prose-sourced date can be `locked` in the very same timeline where a `{ circa: ... }`
  placeholder beside it stays `open`.
- **Places and objects** — each described property (a wall color, a casing color, a
  capacity, an address) locks when prose or a bible renders it, and stays open when it is
  deduced to furnish the world.

Lock state is **orthogonal to focus** (§8): focus governs how vividly the *prose* renders
an entity; lock state governs how firmly each *fact* is held. A `blur`-focus room the
reader barely notices can carry a deep `open` dossier; a single `crisp`-focus object locks
only the few properties the prose actually paints.
